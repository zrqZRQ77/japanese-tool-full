# UI Audit - 2026-07-02

Local URL: `http://localhost:5174/index.html`

## Scope

- Desktop viewport: default browser size, about `1280 x 720`.
- Mobile viewport: `390 x 844`.
- Covered flows: initial load, reading import, word detail, add to vocab, vocab page, flashcard review, typing practice, history page, delete confirmation.

## Confirmed Working

- Page loads through local static server without app console errors.
- Reading import from the reading page works.
- Highlighted words render as clickable `ruby.w` nodes.
- Word detail panel opens for dictionary terms.
- Adding a highlighted word to vocab works.
- Vocab list displays the added item.
- Flashcard review opens and flips.
- Typing practice loads with real prompts and accepts an answer.
- Learning history records the typing practice result.
- Delete actions use the in-app confirmation modal.
- No browser-native `confirm()` dialog appeared during the tested flow.

## Must Fix

### Mobile navigation/layout is broken

Screenshot: `12-mobile-vocab.png`

At `390 x 844`, the sidebar/nav collapses into a very narrow, overlapping vertical stack. Text and icons overlap, and the main content is effectively pushed out of practical view. This is a real usability blocker on phones.

Suggested direction:

- Add a deliberate mobile navigation pattern, such as a top bar plus slide-out menu, or a bottom tab bar.
- Avoid reusing the desktop sidebar layout at phone width.
- Re-test at `390 x 844` and `430 x 932`.

Status: fixed in cache version `20260702-05`.

Follow-up screenshots:

- `22-mobile-390-final.png`
- `23-mobile-430-final.png`
- `24-desktop-after-mobile-fix.png`

## Should Fix

### Desktop full-page screenshots crop right-side content visually

Screenshots: `02-reading-desktop.png`, `16-flashcard-correct.png`

The tested DOM did not report horizontal overflow, but the captured viewport often shows the right side of wide content outside the visible frame. This may be acceptable if the user can scroll within the page, but it should be checked manually in a normal browser window.

Suggested direction:

- Verify whether `.app-content` or inner cards are wider than the viewport after the fixed sidebar.
- Check long reading text and flashcard stage at widths from `1024` to `1280`.

### First-run/persisted state affects test path

After the first successful run, the app no longer shows the hero input because `hasUsedApp` persists. That is expected behavior, but test scripts need to explicitly clear or account for local storage before first-run checks.

Suggested direction:

- Add an audit setup step that clears only this app's local storage.
- Keep separate tests for first visit and returning user.
