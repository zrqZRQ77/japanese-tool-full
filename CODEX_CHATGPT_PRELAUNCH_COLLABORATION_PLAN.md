# Yomeru 上线前协作执行规划（方案 A）

> 适用项目：`/Users/zhouruoqi/Downloads/japaneselearning`
> 适用阶段：MVP 上线前审查、测试、修复、回归与上线判断
> 协作角色：本地 Codex（执行与修改）＋ ChatGPT 网页端（只读审查，仅可在专用目录新建审查报告）
> 核心原则：一个正式文件只能有一个修改者。本地 Codex 是唯一代码修改者和最终结果记录者。

## 0. 快速执行版（日常按此推进）

以下四个阶段是本项目的日常执行主线。后续章节保留完整的权限、问题管理、冲突处理和证据规则，遇到异常或争议时再查阅。

### 阶段一：建立基线

本地 Codex先记录：

```text
基线时间：
当前分支：
当前提交号：
工作区是否有未提交修改：
未提交修改文件列表：
本轮实际审查文件：
最新构建和测试命令、时间及结果：
自动审计结果目录：
视觉截图版本：
当前前端缓存版本：
```

基线有两种方式：

- **只读基线记录（默认）**：记录上述信息，不产生 Git 提交。适合当前修改较多、归属复杂或仍有实验内容的工作区。
- **Checkpoint 提交（可选）**：只有在修改归属清楚、不含无关文件、没有需保留但暂不提交的实验内容，并且用户知道提交范围时才可建立。

不得为了建立基线而机械提交全部未提交内容。建立 checkpoint 前必须先检查 Git 状态并确认文件归属；无法确认时采用只读基线。

### 阶段二：并行检查

- 本地 Codex：运行构建、自动化、核心流程、PDF、导出、存储、安全、异常和视觉检查。
- ChatGPT 网页端：只读审查产品逻辑、测试覆盖、文案、隐私、代码风险、文档一致性和截图，并在 `.ai-coordination/reviews/` 新建本轮报告。
- 双方基于同一基线工作；网页端除新建自己的审查报告外不修改任何文件，也不把未运行项目标记为 PASS。

### 阶段三：修复与回归

本地 Codex负责复现、定级、去重、修改和验证，并维护唯一问题队列。优先清理 P0/P1，再处理影响核心体验的 P2；P3 默认进入上线后清单。

网页端在修复期间不审查变化中的文件，等待本地 Codex提供新的复核基线。

### 阶段四：最终复核

本地 Codex完成完整回归并提供最新基线、测试结果、截图和未完成项目。网页端只读复核证据与文档是否一致，并新建最终复核报告。本地 Codex据此更新正式报告，并给出允许上线、带条件上线或暂缓上线的结论。

### 三层证据规则

| 证据层级 | 可以确认 | 不能据此推断 |
|---|---|---|
| 静态代码与文档审查 | 产品逻辑、入口一致性、文案、配置、明显代码风险 | 实际点击、文件导出、声音播放或浏览器兼容性已经通过 |
| 本地运行与自动化测试 | 当前本地测试环境中的点击流程、存储、PDF、导出、错误处理和自动化结果 | Safari、iPhone、微信浏览器或线上环境一定表现相同 |
| 真实设备与线上验收 | Safari、iPhone、微信浏览器、实际 TTS 音色、首次网络加载、CDN、Vercel、缓存和移动端下载行为 | 未覆盖设备或网络环境中的表现 |

所有 PASS 必须注明证据层级。自动化通过不能替代真实设备或线上验收；本地 Chrome 中通过，只能证明当前本地测试环境内通过。

### 本轮停止条件

> 当未解决的 P0/P1 清零，核心流程回归测试通过，正式文档与当前代码一致，并且所有未完成的真实设备和线上项目已明确标记为 `PENDING` 后，停止新增功能和非阻断性视觉优化。本轮 P2/P3 统一进入上线后清单，项目转入真实设备、线上环境和首次用户验收。

达到停止条件后，任何新增功能或非阻断视觉调整都必须作为新任务处理，不能继续扩大本轮上线检查范围。

“停止条件”用于冻结本轮开发范围，达到后停止新增功能和非阻断优化；“完成定义”用于判断整个上线前协作是否正式结束。达到停止条件后，仍需完成最终复核，并将未完成的真实设备或线上验收如实保留为 `PENDING`。

### 快速执行版与详细阶段对应关系

| 快速执行阶段 | 对应详细阶段 |
|---|---|
| 阶段一：建立基线 | 详细阶段 0 |
| 阶段二：并行检查 | 详细阶段 1～2 |
| 阶段三：修复与回归 | 详细阶段 3～4 |
| 阶段四：最终复核 | 详细阶段 5～6 |

快速执行版是日常工作主线；详细阶段用于明确具体分工、证据要求和异常处理，两者不是两套独立流程。

## 1. 目标与最终产出

本轮协作的目标是验证 Yomeru 当前 MVP 是否满足上线条件，发现并修复阻断问题，完成回归测试，并给出以下三种结论之一：

- 允许上线
- 带条件上线
- 暂缓上线

“带条件上线”表示本地技术检查已通过，未完成的真实设备、线上环境或其他外部验收项目已经明确记录，并经判断不构成已知 P0/P1 阻断；上线范围、待验收项目和回退条件均已明确。“完成修复后再判断”不属于上线结论，应保持为暂缓上线。

最终应产生：

1. 经过验证的前端代码与必要测试脚本。
2. 更新完成的 `PRE_LAUNCH_TEST_REPORT.md`。
3. 与当前代码一致的 `PRE_LAUNCH_CHECKLIST.md`。
4. 清晰记录的未解决问题和上线后优化项。
5. 最终上线结论及其测试证据。

## 2. 单一修改者规则

### 2.1 本地 Codex 的权限

本地 Codex可以：

- 读取整个项目。
- 运行项目已有的构建、检查、自动化测试和视觉验证命令。
- 启动本地静态服务器并检查页面。
- 修改 `frontend/`、`backend/`、`scripts/` 和测试工具中的文件。
- 修复已经确认的问题。
- 更新正式检查清单和测试报告。
- 在修改后执行回归测试。
- 汇总网页端建议，并决定接受、调整或拒绝。

本地 Codex不可以：

- 未经用户授权修改生产环境、域名、付费、账号或外部服务配置。
- 为了“顺便优化”而扩大产品功能范围。
- 删除用户已有代码、历史资料或未确认用途的文件。
- 仅凭静态阅读把测试项目标记为真实设备 PASS。
- 把未经验证的网页端建议直接写成 PASS 结论。

### 2.2 ChatGPT 网页端的权限

ChatGPT 网页端可以：

- 读取项目代码、文档、测试报告和截图。
- 审查产品闭环、测试覆盖、文案、隐私、安全、可访问性和视觉一致性。
- 指出矛盾、遗漏、风险和可能的修复方向。
- 根据本地 Codex提供的结果进行独立复核。
- 输出结构化问题清单和建议。
- 仅在 `.ai-coordination/reviews/` 中新建自己的审查报告。

ChatGPT 网页端不可以：

- 修改任何正式项目文件。
- 修改 `frontend/`、`backend/`、测试脚本或部署配置。
- 更新 `PRE_LAUNCH_TEST_REPORT.md`、`PRE_LAUNCH_CHECKLIST.md` 的正式状态。
- 执行 Git 提交、合并、回退、删除或覆盖操作。
- 编辑或删除任何已有审查报告，包括自己上一轮创建的报告。
- 修改 `.ai-coordination/` 中除新建 `reviews/chatgpt-review-*.md` 外的任何文件。
- 把没有实际运行过的测试标记为 PASS。
- 在本地 Codex修复期间基于旧版本提出“已经修复”的结论。

网页端对项目保持只读，唯一写入例外是在 `.ai-coordination/reviews/` 中新建审查报告。所有修改建议都交给本地 Codex实施。若需更正旧报告，必须新建 `chatgpt-review-日期-round-轮次-correction-序号.md`，不得覆盖原报告。

## 4. 共同工作基线

每轮审查开始前，本地 Codex应提供一个明确基线：

- 审查日期和时间。
- 当前 Git 分支和提交号（如果仓库已提交）。
- 工作区是否存在未提交修改。
- 未提交修改文件列表，以及本轮实际审查文件。
- 最近一次构建和自动检查的命令、执行时间与结果。
- 自动审计结果目录和视觉截图版本。
- 当前前端缓存版本。

推荐记录格式：

```text
npm test｜2026-07-11 14:20｜PASS
cd frontend && npm run verify:flows｜2026-07-11 14:26｜PASS
真实 Safari｜尚未执行｜PENDING
```

默认使用只读基线记录，不要求创建 Git 提交。只有在修改归属、提交范围和用户知情均明确时才允许建立 checkpoint 提交；不得把无关修改、测试产物或实验内容机械地一起提交。

网页端必须基于同一基线审查。若本地代码在网页端审查过程中发生变化，网页端的问题清单需注明“基于旧基线”，由本地 Codex重新确认后才能进入修复队列。

## 5. 统一问题记录格式

两个角色都使用以下格式，避免重复和结论不一致：

```text
编号：AREA-001
来源：本地 Codex / ChatGPT 网页端 / 用户
基线：分支或提交号＋日期
等级：P0 / P1 / P2 / P3
状态：NEW / CONFIRMED / FIXING / FIXED / VERIFIED / REJECTED / DEFERRED
涉及文件：
发现：
复现方式或证据：
证据层级：静态代码与文档审查 / 本地运行与自动化测试 / 真实设备与线上验收
证据位置：测试命令、报告路径、截图目录或真实设备记录
预期结果：
建议：
是否阻止上线：是 / 否 / 待确认
处理结论：由本地 Codex填写
```

编号前缀建议：

- `CORE`：核心学习流程
- `DATA`：本地数据、备份与恢复
- `PDF`：PDF 导入
- `EXPORT`：PPTX、PNG、JPEG 导出
- `SEC`：安全与输入处理
- `ERR`：异常与降级
- `UI`：视觉和响应式
- `A11Y`：可访问性
- `PERF`：性能和资源加载
- `DOC`：文档或文案一致性
- `DEPLOY`：构建、缓存和部署

P0/P1 问题必须有复现证据、修复记录和回归结果。仅有建议而没有证据的问题保持 `NEW`，不能直接视为上线阻断项。

## 6. 工作阶段与分工

### 阶段 0：冻结范围和建立基线

本地 Codex：

1. 查看 Git 状态，保护用户已有修改。
2. 记录当前基线。
3. 核对 MVP 边界：粘贴文本是核心输入；文字型 PDF 为 Beta；公开 MVP 不依赖后端。
4. 核对正式测试资料是否与当前代码一致。

网页端：

1. 阅读本规划和四份核心资料。
2. 确认只读角色及唯一的审查报告新建权限。
3. 等待本地 Codex提供基线，不提前把历史报告视为当前事实。

阶段完成标准：双方基于同一版本工作，公开 MVP 范围没有歧义。

### 阶段 1：并行检查

本地 Codex负责可执行验证：

1. 构建与静态检查。
2. JavaScript、HTML、必要资源和缓存版本检查。
3. Playwright 核心流程审计。
4. 视觉和响应式验证。
5. PDF、导出、localStorage、异常输入和 XSS 测试。
6. 检查控制台错误、资源 404 和失败降级。

优先使用项目已有命令：

```bash
npm test
cd frontend && npm run verify:fast
cd frontend && npm run verify:flows
cd frontend && npm run verify:visual
cd frontend && npm run verify:all
cd backend && npm test
```

是否运行后端测试由本地 Codex根据改动范围决定；后端不是公开 MVP 的运行依赖。

网页端负责独立只读审查：

1. 产品闭环和首次使用路径。
2. 测试清单是否遗漏关键边界。
3. 中文、日文、英文文案及术语一致性。
4. 隐私说明是否符合浏览器本地存储的真实行为。
5. 安全、可访问性和错误提示风险。
6. 根据截图检查视觉层级、遮挡、空状态和移动端可理解性。
7. 检查三份上线资料之间是否存在过期或相互矛盾的结论。
8. 按 `.ai-coordination/reviews/REVIEW_TEMPLATE.md` 新建本轮审查报告，不覆盖旧报告。

阶段完成标准：本地测试结果和网页端建议分别形成清单，尚不混写结论。

### 阶段 2：问题汇总与去重

本地 Codex是汇总负责人：

1. 把网页端建议逐条映射到统一问题编号。
2. 复现网页端提出的代码问题。
3. 合并重复问题。
4. 对无法复现的问题标记 `REJECTED` 或保留 `NEW`，写明理由。
5. 按 P0、P1、P2、P3 排序。

处理优先级：

1. P0：立即停止其他修复并处理。
2. P1：上线前必须处理或明确取消相关公开能力。
3. P2：优先处理影响核心体验或数量较多的问题。
4. P3：记录为上线后优化，不应无边界扩展本轮工作。

阶段完成标准：形成唯一问题队列，每个问题有负责人、等级和状态。

### 阶段 3：本地 Codex统一修复

本地 Codex：

1. 从最高等级问题开始，小步修改。
2. 每次修复尽量只覆盖一个问题或一个强相关问题组。
3. 保留现有产品边界和业务逻辑，除非问题本身要求调整。
4. 修改后立即运行与改动风险相称的定向测试。
5. 记录修改文件、验证命令和结果。

网页端：

- 此阶段不修改文件。
- 可以解释建议或补充证据。
- 不重复审查正在变化的文件；等待本地 Codex发出复核基线。

阶段完成标准：所有 P0/P1 已修复或通过产品收口消除；关键 P2 有明确处理结论。

### 阶段 4：完整回归

本地 Codex至少验证：

- 首页可正常打开，无白屏。
- 粘贴日语文本能够进入阅读流程。
- 点击词语、收藏、生词本、复习、练习和历史形成闭环。
- 刷新后关键本地数据符合预期。
- 清除、备份和恢复数据行为正确。
- 文字型 PDF 能处理；扫描、损坏、加密或过大 PDF 有清楚降级。
- PPTX、PNG、JPEG 导出与公开承诺一致。
- 用户输入不会作为 HTML 或脚本执行。
- 手机和桌面关键尺寸没有阻断性遮挡。
- 构建产物包含必要资源。

真实 Safari、iPhone Safari、微信内置浏览器等无法被本地自动化完全代替的项目，必须标记为 `PENDING` 或由用户完成真实设备验收，不能推测为 PASS。

阶段完成标准：没有未解决的 P0/P1；核心自动测试通过；真实设备待办被明确列出。

### 阶段 5：网页端独立复核

本地 Codex向网页端提供：

- 最新基线。
- 修改摘要。
- 测试结果。
- 关键页面截图或视觉审计结果。
- 仍未解决的问题。

网页端复核：

1. 原建议是否被正确处理。
2. 修复是否引入新的产品或文案矛盾。
3. 正式报告中的 PASS 是否有证据。
4. PENDING、DEFERRED 和真实设备项目是否被如实保留。
5. 最终上线结论是否与证据一致。

网页端仍只提供建议，只能新建最终复核报告，不直接修改正式报告或已有审查报告。

### 阶段 6：最终报告和上线判断

本地 Codex统一更新：

- `PRE_LAUNCH_TEST_REPORT.md`
- `PRE_LAUNCH_CHECKLIST.md`
- 必要时更新 `DEPLOYMENT.md`
- 必要时清理 `MVP_LOGIC_CLOSURE_REVIEW.md` 中已经过期或自相矛盾的历史结论

最终结论规则：

- 存在 P0：暂缓上线。
- 存在未解决 P1：暂缓上线或取消对应公开能力后重新验证。
- 无 P0/P1，核心闭环通过，但仍有少量非阻断 P2：可以上线并记录优化计划。
- 本地技术检查已通过，未完成的真实设备、线上环境或其他外部验收项目已明确记录，且不构成已知 P0/P1 阻断，同时上线范围、待验收项目和回退条件均明确：可以判定为带条件上线。

## 8. 冲突与异常处理

- 如果发现网页端修改了文件，立即暂停合并；本地 Codex先查看差异，再由用户决定是否保留。不得直接覆盖。
- 网页端在允许目录外修改文件，或编辑、删除已有审查报告，均视为越权修改。
- 如果双方结论不同，以可复现测试和当前代码为准；文案或产品判断由用户最终决定。
- 如果测试报告与代码不一致，报告状态退回 `PENDING`，重新验证。
- 如果工作区存在用户未提交修改，本地 Codex必须保留并避免覆盖；无法安全区分时先报告用户。
- 如果需要外部账号、生产部署、域名、分析平台或真实设备权限，先列为外部依赖，不擅自操作。
- 不通过复制项目、覆盖文件或重置 Git 来解决冲突。

## 9. 建议节奏

建议按以下检查点同步，而不是两边持续读取同一变化中的版本：

1. 基线确认后：网页端开始第一轮只读审查。
2. 本地自动检查完成后：交换第一批问题清单。
3. P0/P1 修复完成后：本地发布新的复核基线。
4. 完整回归完成后：网页端进行最终复核。
5. 正式报告更新后：用户确认真实设备和上线安排。

每次同步只传递新增结果和状态变化，避免重复发送完整代码或全部报告。
