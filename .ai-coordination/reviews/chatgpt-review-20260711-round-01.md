### TEST-002：自动审计 PASS 不能证明 Kuromoji 已实际加载和参与分词

- 建议等级：P2
- 证据层级：静态代码与文档审查
- 涉及文件：`frontend/tools/ui-audit.mjs`、`frontend/app.js`、`frontend/audit-screenshots/2026-07-11T07-51-55-692Z/ui-audit-report.json`
- 发现：第一轮基线的 UI 审计能证明输入后生成了带假名的阅读结果，但报告没有记录实际 tokenizer mode。前端存在 Kuromoji 加载失败后退回本地词典匹配的路径，因此“reading import PASS”只能证明页面产生结果，不能证明远程 Kuromoji 词典已成功下载、构建并用于分词。
- 证据位置：`2026-07-11T07-51-55-692Z/ui-audit-report.json` 的 reading 状态只记录输出内容，未记录 tokenizer mode；`frontend/app.js` 中 Kuromoji 与 fallback 两条渲染路径。
- 可能影响：测试报告可能把轻量词典 fallback 的成功误写成 Kuromoji 功能通过，掩盖首次加载、网络、性能或词典构建问题。
- 建议：由页面暴露明确的 tokenizer mode，并在审计报告中记录；若公开承诺 Kuromoji，则测试必须断言模式为 `kuromoji`，同时单独测试加载失败后的 fallback。若当前产品只承诺本地轻量词库，应按真实模式表述，不能借该 PASS 证明远程 Kuromoji 可用。
- 是否必须实际运行确认：是
- 是否可能阻止上线：待确认

### DEPLOY-001：根目录构建产物漏掉页面引用的 analytics.js

- 建议等级：P2
- 证据层级：静态代码与文档审查
- 涉及文件：`scripts/build-frontend.mjs`、`frontend/index.html`、`README.md`、根目录 `vercel.json`
- 发现：第一轮基线的 `frontend/index.html` 引用了 `analytics.js`，但根目录构建脚本的手工文件清单未包含该文件。按照 Vercel 根目录构建并发布 `dist/` 时，页面会请求一个未进入产物的本地脚本。
- 证据位置：`frontend/index.html` 的 `analytics.js` script 标签；第一轮基线 `scripts/build-frontend.mjs` 的文件复制清单；根目录 `vercel.json` 的 `outputDirectory: dist`。
- 可能影响：线上首次访问出现资源 404 或资源受限提示；即使分析功能未启用，也会造成部署产物与页面引用不完整，降低首屏可信度。
- 建议：把 `analytics.js` 纳入构建产物，并在构建阶段自动检查 `index.html` 引用的所有本地 CSS、JavaScript、SVG 和数据文件是否真实存在，避免继续依赖易遗漏的手工清单。
- 是否必须实际运行确认：是
- 是否可能阻止上线：待确认

### DATA-001：水平测试重置绕过 safeStorage 直接操作 localStorage

- 建议等级：P2
- 证据层级：静态代码与文档审查
- 涉及文件：`frontend/app.js`
- 发现：第一轮基线的 `resetLevelTest()` 直接调用 `localStorage.removeItem('reading_level_result')`，而项目其他位置已经提供带异常捕获的 `safeStorage` 封装。该调用绕过了统一的存储失败处理。
- 证据位置：第一轮基线 `frontend/app.js` 中 `safeStorage` 定义和 `resetLevelTest()` 实现。
- 可能影响：在隐私模式、存储被禁用或浏览器抛出访问异常时，重置动作可能中断，导致水平测试 UI 与存储状态不一致。
- 建议：改用 `safeStorage.removeItem()`，并增加 localStorage 被禁用、读取损坏和容量异常时的定向测试，确认页面不会白屏且能给出可理解反馈。
- 是否必须实际运行确认：是
- 是否可能阻止上线：否
