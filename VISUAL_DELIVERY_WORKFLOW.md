# Yomeru 页面视觉交付流程

适用范围：Yomeru 网站内所有会改变用户可见页面、布局、响应式、文字层级或交互状态的修改。

## 默认执行顺序

```text
读取项目规则与当前交接记录
→ 核验 Git 和现有实现
→ 完成代码修改
→ 运行受影响的功能测试与维护门禁
→ 生成桌面、窄桌面和手机截图
→ 逐张检查实际页面
→ 发现问题时继续修改并重新截图
→ 确认无明显问题后才向用户提供 Preview 或截图
```

## 截图检查范围

至少覆盖：

- 桌面宽屏；
- 窄桌面或平板宽度；
- 手机宽度；
- 本轮实际改变的关键状态，例如初始页、操作中、展开状态、错误状态和结果页。

## 必查项目

逐张检查：

- 核心内容是否完整可见；
- 是否存在横向溢出、裁切、遮挡或文字重叠；
- 卡片宽高、留白和视觉重心是否协调；
- 标题、正文、按钮和辅助文字的层级是否清楚；
- 同类按钮、标签和卡片的尺寸与文案是否统一；
- 桌面、窄屏和手机断点是否在合适的宽度切换；
- 输入框、弹窗、键盘和固定元素是否遮挡主要操作；
- 页面是否出现开发文案、内部说明或不必要的次级信息。

## 交付规则

- 自动测试通过不等于视觉验收通过。
- 未完成截图检查时，不向用户声称页面已经完成。
- 截图中发现的问题直接修复，不等待用户再次指出同类问题。
- 最终回复只提供已经通过内部截图检查的候选。
- Production、数据库、环境变量和其他高风险操作仍需单独明确授权。

## 电车挑战当前截图命令

```bash
cd frontend
npm run test:train-static
npm run visual:train
```

截图默认输出到 `frontend/audit-screenshots/train-challenge-latest/`。截图目录属于本地检查产物，不进入功能提交。
