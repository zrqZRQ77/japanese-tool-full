# Yomeru 项目专属设计系统

## 0. 文件入口

当前项目的设计规范分三层：

| 文件 | 作用 |
| --- | --- |
| `DESIGN.md` | 文字版设计准则，给人和 AI 看。新增页面、组件、视觉调整都先遵守这里。 |
| `frontend/design-system.css` | 当前网站真正生效的设计 token 与组件样式，是代码里的主设计系统文件。 |
| `YOMERU_UI_KIT.md` | 当前可执行的精简设计规范，记录字体、字号、颜色、组件和组合规则。 |
| `frontend/yomeru-ui-kit.html` | 可直接打开的视觉参考页，用来查看颜色、排版、按钮、输入框、卡片和完整组件组合。 |

正式入口约束：

- `frontend/index.html` 的正式 CSS 顺序固定为：`styles.css` → `design-system.css` → `grammar-layout.css` → `typography.css`。
- `typography.css` 最后加载，负责中文、日文、日语学习内容的字体、字号、行高和字重最终规则。
- 新增或稳定下来的通用视觉规则进入 `frontend/design-system.css`；语法页布局进入 `grammar-layout.css`；不要新增临时覆盖层。
- 运行时补丁脚本只能作为迁移期工具；当 DOM 结构已经正式落地到 `index.html` / `app.js` 后，应移除入口引用。

辅助旧文件：

- `frontend/styles.css` 保留基础布局与历史兼容；正式通用组件规则以 `frontend/design-system.css` 为准。
- `frontend/grammar-layout.css` 只承载语法页布局，不重新发明颜色、字体和通用组件。
- `frontend/typography.css` 是字体与字号的唯一正式来源。
- `frontend/design-reference.html` 为历史参考；当前统一使用 `frontend/yomeru-ui-kit.html`。

外部参考：

- Claude design.md：<https://getdesign.md/claude/design-md>
- Material Design 3：<https://m3.material.io/>

本文件是 Yomeru 网页设计与前端实现的唯一设计准则。新增页面、组件、样式和视觉调整，都应先遵守这里的规则，再考虑单个页面的特殊处理。

一句话定位：

> Yomeru 是一个安静、清晰、适合长期学习的日语阅读工作台。整体保留紫色 Material 3 token，但版式、留白、组件克制感和内容优先级参考 Claude 的干净编辑型界面。

## 1. 设计目标

Yomeru 要解决的问题不是“某个按钮好不好看”，而是“所有按钮、卡片、输入区、阅读区和练习模块放在一起时，是否像同一个产品”。

### 核心原则

1. 阅读正文永远是视觉主角。
2. 紫色是品牌主色，不改成蓝绿、橙色或其他主题。
3. Claude 式规则负责气质：留白大、文字清楚、装饰少、组件安静。
4. 页面使用 Material 3 的语义 token，不在局部随手写颜色。
5. 布局使用 8px 节奏，建立稳定的排版秩序。
6. 组件先标准化，再组合页面。
7. 页面不要卡片套卡片，不要装饰性渐变、光斑、玻璃拟态或复杂阴影。
8. 学习状态颜色只表达含义，不做装饰。
9. 移动端、键盘焦点、触控目标和文本重排必须一起考虑。

### 参考体系的分工

| 来源 | Yomeru 借鉴什么 | 不照搬什么 |
| --- | --- | --- |
| Claude | 温和编辑感、大留白、内容优先、少装饰、低阴影、清楚的单一主行动 | 不照搬陶土色/暖橙品牌色 |
| Material Design 3 | 紫色 token、surface 层级、状态反馈、按钮层级、导航与弹窗结构 | 不照搬过强的默认组件感 |
| Ant Design | 表单秩序、信息密度、网格对齐、后台工具式页面组织 | 不照搬企业后台的冷感和拥挤感 |
| Apple HIG | 克制、留白、阅读优先、少装饰、清晰层级 | 不追求 iOS 风格拟态 |

## 2. Claude 式 Yomeru 视觉方向

Claude 的可借鉴部分不是颜色，而是“像一份好读的编辑稿一样组织界面”：少一点装饰，多一点呼吸感；少一点组件炫技，多一点稳定层级；让用户自然看见主任务，而不是被按钮、卡片和色块推着走。

### 迁移到 Yomeru 的规则

1. **单列优先**：核心流程优先使用一条清楚的阅读/操作路径，只有信息密度真正需要时才拆成多列。
2. **浅表面优先**：用 `surface`、边框和留白建立结构，少用阴影。
3. **主色克制**：紫色只出现在 Logo、品牌名、主 CTA、选中态、焦点和少量学习语义标注，不用于大面积背景。
4. **文案减法**：能由结构表达的说明不再额外写一行文字。
5. **流程连续**：多个模块之间要像同一个任务流，而不是三个独立广告卡片。
6. **编辑感排版**：中文标题、正文和控件保持清晰 sans；日语阅读正文、词语与语法例句使用 Mincho，行高宽松。
7. **静态稳定**：hover 可以轻微反馈，但不要上浮、放大、强阴影。
8. **组件同温层**：同一屏里的卡片、输入框、工具栏要共享同一种边框、圆角和 surface，不出现“某个局部特别亮”的断层。

### 首页的 Claude 式目标

首页不是营销介绍，而是一块安静的导入工作台：

- Logo 属于站点身份，优先放在首屏左上角品牌位；首页首屏只显示图标，不和主标题混在一起，也不在图标旁重复品牌名。图标默认不画明显边框，只有在复杂背景上才允许加轻量容器。
- 标题短，少解释。
- 输入区是主角。
- 次要入口只用文字按钮。
- 功能预览在首屏之后出现，是两列等高竖向卡片，而不是横向宣传卡。
- 预览里的紫色使用浅紫，不能和主按钮争抢。

## 3. 品牌视觉方向

Yomeru 的视觉气质是“阅读工作台”，不是营销落地页，也不是游戏化学习应用。

### 应该呈现

- 干净、安静、可信。
- 紫色主调明确，但不大面积染紫。
- 大面积背景优先使用中性白、微灰或纸面色；紫色是品牌强调，不是页面底色。
- 输入、阅读、复习之间有一致的组件语言。
- 页面有编辑型留白，允许“少一点内容”，不把所有能力塞满首屏。
- 页面密度适中，适合反复打开和长时间学习。
- 日语正文、假名、词义和练习反馈清楚可扫读。

### 紫色使用边界

紫色应该让用户记住品牌和当前状态，而不是让整页都带颜色。

应该使用紫色：

- Logo 和品牌名称。
- 主按钮和关键行动。
- 当前导航、当前工具和焦点状态。
- 轻量学习语义标注、状态点和小标签。

不应该使用紫色：

- 长时间阅读的页面大背景。
- 正文、词汇主标题、表格正文和列表主信息。
- 普通卡片边框、普通筛选按钮、大面积信息容器。
- 同屏多个互相竞争的填充色块。

内容密集页面的主文字优先使用深灰或近黑，辅助信息使用中性灰。紫色只用于提示“这里被选中、这里可以行动、这里属于品牌”。

### 应该避免

- 每个模块各自漂亮，但放一起像不同产品。
- 大面积彩色卡片。
- 过多阴影和悬浮层。
- 超大营销型 hero。
- 为了解释功能而堆叠过多文字。
- 多个视觉中心同时抢注意力。
- 圆角过大、胶囊过多。
- 用红色表示普通复习压力。
- 随手新增一个紫色、灰色、字号或间距。

| 错误 | `#BA1A1A` | `#FFDAD6` | 导入失败、删除确认、系统错误 |

### 颜色使用规则

1. 主流程只能有一个紫色主按钮。
2. 阅读正文不要大面积染色，词语标注只用柔和浅底。
3. `error` 只给真正错误，学习压力和“需复习”使用 `warning`。
4. 边框优先用 `outline-variant`，不要用深色边框制造噪音。
5. 新增颜色前，先判断它属于 primary、secondary、surface、warning 还是 error。
6. 如果无法归类，就不要新增颜色。

## 5. 间距系统

Yomeru 使用 8px 网格作为整体节奏。允许 `4px` 和 `12px` 作为微调，但页面级布局必须落在稳定阶梯上。

| Token | 值 | 用途 |
| --- | --- | --- |
| `--space-1` | `4px` | 极小间距、图标内部修正 |
| `--space-2` | `8px` | 图标和文字、紧凑控件 |
| `--space-3` | `12px` | 按钮组、表单小间距 |
| `--space-4` | `16px` | 卡片内边距、模块内分组 |
| `--space-5` | `24px` | 标准区块间距 |
| `--space-6` | `32px` | 阅读区内边距、主模块内边距 |
| `--space-7` | `48px` | 页面大区块 |
| `--space-8` | `64px` | 首屏或大段落分隔 |

### 间距规则

1. 页面外边距优先使用 `24 / 32 / 48`。
2. 卡片内边距优先使用 `16 / 24 / 32`。
3. 按钮组和表单项间距优先使用 `8 / 12 / 16`。
4. 同一页面内，同等级模块必须使用同一间距。
5. 间距不够时，先调整布局层级，不要随手写 `19px`、`23px` 这类值。
6. Claude 式页面允许更大的空白；如果页面显得散，先收窄内容宽度，不要急着填内容。

## 6. 字体排印

字体系统以 `frontend/typography.css` 为唯一正式来源。

| Token | 值 | 用途 |
| --- | --- | --- |
| `--type-font-ui-zh` | `Inter`, `PingFang SC`, `Microsoft YaHei`, `Noto Sans SC`, sans-serif | 中文导航、按钮、表单和说明 |
| `--type-font-ui-ja` | `Noto Sans JP`, `Hiragino Sans`, `Yu Gothic`, `Meiryo`, sans-serif | 日文小控件、导航和短标签 |
| `--type-font-content-ja` | `Noto Serif JP`, `Hiragino Mincho ProN`, `Yu Mincho`, serif | 日语正文、词语详情和语法例句 |

### 核心字号阶梯

| Token | 值 | 用途 |
| --- | --- | --- |
| `--type-size-ruby` | `10px` | 假名 |
| `--type-size-caption` | `12px` | 标签、元信息 |
| `--type-size-meta` | `13px` | 辅助说明 |
| `--type-size-control` | `14px` | 按钮、输入、筛选 |
| `--type-size-body` | `15px` | UI 正文 |
| `--type-size-subtitle` | `18px` | 面板标题 |
| `--type-size-section-title` | `20px` | 区块标题、阅读正文 |
| `--type-size-page-title` | `28px` | 页面标题 |
| `--type-size-hero` | `36px` | 首页主标题 |
| `--type-line-ui` | `1.5` | 控件行高 |
| `--type-line-body` | `1.7` | UI 正文行高 |
| `--type-line-reading` | `2` | 日语正文行高 |

### 排印规则

1. UI 正文默认 `15px`，不要一会儿 14 一会儿 16。
2. 卡片和面板标题默认 `18px`，不要使用 hero 级标题。
3. 日语阅读区可以更大、更松，优先保护可读性。
4. 不使用负字距。
5. 不用视口宽度动态缩放普通 UI 字号。
6. 文本必须在按钮、卡片和面板内完整显示，不能挤出或遮挡。
7. 首页和关键页面标题可以有编辑感，但不能像营销页那样过度巨大。
8. 辅助说明能删就删，保留的说明必须帮助用户做下一步动作。

## 7. 圆角与阴影

Yomeru 的形状语言偏克制，圆角用于降低硬度，不用于制造可爱感。

| Token | 值 | 用途 |
| --- | --- | --- |
| `--radius-xs` | `3px` | 正文标注、技术细节 |
| `--radius-sm` | `5px` | 标签、图标按钮 |
| `--radius-md` | `7px` | 按钮、输入框 |
| `--radius-lg` | `8px` | 卡片、阅读面板、模块容器 |
| `--radius-xl` | `8px` | 历史兼容，不用于新设计 |
| `--radius-pill` | `999px` | 只用于真正需要胶囊形的状态 |

### 形状规则

1. 普通卡片最大圆角 `8px`。
2. 不要大量使用胶囊按钮。
3. 图标按钮保持稳定正方形点击区域。
4. 阴影只表达层级和浮层，不做装饰。
5. 页面区块不要全部漂浮成卡片。
6. Claude 式默认是“边框 + 留白”，阴影是最后才使用的手段。

## 8. 组件契约

组件契约是最重要的一层。页面只能组合标准组件，不应该每个区域重新发明按钮、输入框和卡片。

### 标准组件

| 组件 | 类名 | 用途 |
| --- | --- | --- |
| 主按钮 | `.btn-primary` | 当前流程的主要动作 |
| 次按钮 | `.btn-secondary` | 辅助动作 |
| 文字动作 | `.btn-secondary.btn--text` | 低强调操作 |
| 图标按钮 | `.btn-icon` 或对应 icon button 类 | 工具按钮、关闭、删除 |
| 卡片 | `.card` + `.card-pad` | 独立工具、重复列表项、弹窗内容 |
| 阅读面 | `.reading-output` | 日语文章、假名和词语标注 |
| 输入区 | `input`, `textarea`, `.typing-input`, `.grammar-search` | 表单和练习输入 |
| 导入面板 | `.source-dropzone` | 粘贴、上传、链接导入 |
| 工具栏 | `.reader-toolbar` | 阅读工具、字号、翻译、朗读 |
| 主导航 | `.app-sidebar`, `.sidebar-nav`, `.nav-item` | 桌面左侧导航、移动端底部导航 |
| 浮层 | `.reader-tool-popover`, `.menu-panel`, `.detail-drawer` | 上下文工具和详情 |
| 标签 | `.typing-chip`, `.vocab-practice-tag`, `.gpoint-level` | 学习状态和分类 |

### 按钮规则

1. 每个可见流程只能有一个主按钮。
2. 按钮触控目标至少 `44px` 高。
3. 主按钮使用紫色 filled。
4. 次按钮使用白底或浅紫 tonal。
5. 低优先级操作使用文字按钮。
6. 危险操作不能和普通主按钮长得一样。

### 输入规则

1. 输入框使用 `surface` 或 `surface-container`。
2. 默认 `1px` 边框，focus 使用紫色状态环。
3. 大输入区应该像纸面，而不是像广告卡片。
4. 输入、上传、示例和开始按钮必须在同一个视觉组里。

### 卡片规则

1. 卡片用于独立对象，不用于包裹整个页面。
2. 不允许明显卡片套卡片。
3. 重复列表卡片必须密度一致。
4. 卡片内标题、描述、操作区顺序固定。
5. 阴影越少越好，优先用 surface 层级和边框建立结构。
6. 同一屏里的卡片不要出现一种有重阴影、一种纯边框、一种彩色底的混合状态。
7. 如果卡片只是流程的一步，它应该更像“流程节点”，而不是宣传卡。

### Surface 层级规则

页面不要靠一层层边框证明结构。Yomeru 的二级工作区优先使用“平铺工作台”：

1. **页面级 surface**：只提供宽度、留白和背景，不画明显边框、不加阴影，不作为卡片出现。
2. **模块级 surface**：用于真正独立的工具模块，例如查语法、闪卡复习、设置分组；允许 1px 边框，默认不使用重阴影。
3. **对象级 surface**：用于重复对象或空状态，例如单个资料、单张复习卡、单个收藏项；如果它位于模块级 surface 内，必须弱化为浅底、分割线或无阴影。
4. 表格、筛选栏和列表属于同一个工作流时，应放在同一块平面上，用分割线、表头、行高和留白分区，不再套一张大卡片。
5. 如果一个区域里已经有明显小卡片，外层容器必须退成透明或平面，不再同时出现边框、圆角和阴影。
6. 空状态不要做成“卡片里的卡片”；优先用居中文字、浅底提示区或虚线区域表达。

### 导航规则

1. 桌面侧边导航使用浅 `surface-container-low`，不使用深色整栏。
2. 选中态使用浅紫 `primary-container`，文字使用 `on-primary-container`。
3. 导航项最小高度 `44px`，图标和文字左对齐，圆角不超过 `8px`。
4. 侧栏宽度以不压缩主内容为前提，默认使用 `224px`。
5. 侧栏不使用重阴影，结构通过右边框、分割线和留白表达。
6. 手机端保留顶部品牌栏，主导航放在底部，底部导航也使用浅 surface、细边框和浅紫选中态。
7. 导航只承载位置和切换，不放宣传文案或复杂学习状态。
8. 设置与帮助入口使用语义图标，不用“设”这类汉字作为图标。
9. 生词本、语法本、练习等二级页面也必须沿用紫色 token，不保留旧绿色主题。

### 下拉菜单与二级菜单

Yomeru 的菜单应该像阅读工具的一部分：轻、清楚、可扫读。菜单浮层默认使用 `surface` 或 `surface-container-low`，边框使用 `outline-variant`，圆角不超过 `8px`，阴影只用于让浮层从页面上分离。

经典样式优先从以下四类中选择：

| 样式 | 适用场景 | 视觉规则 |
| --- | --- | --- |
| 紧凑列表菜单 | 设置、帮助、导入、更多操作等少量命令 | 单列列表，行高 `40-44px`，左侧图标可选，当前项用浅紫底 |
| 分组菜单 | 同一入口下有 2-3 组不同性质的操作 | 使用小标题和分割线，标题为 `text-xs`，不做彩色分组卡 |
| 级联菜单 | 有明确父子层级，例如练习类型、导出格式、阅读工具 | 一级项右侧用箭头提示，二级浮层与一级浮层同高或轻微错位，不使用多层重阴影 |
| 宽面板菜单 | 功能较多但属于一个工作区，例如学习工具、资料库入口 | 允许两列或三列，但每项仍是行式入口；不做营销式 mega menu，不放大图和宣传文案 |

交互规则：

1. 菜单项最小高度 `40px`，触控场景至少 `44px`。
2. 菜单宽度默认 `220-280px`；宽面板不超过 `640px`，手机端改为单列或底部抽屉。
3. 菜单项文案使用动词或名词短语，不写解释句；必要说明放在次级文字，字号 `text-xs`。
4. 选中、hover、focus 都使用浅紫或浅 surface，不使用整块深紫。
5. 危险操作必须放在分组末尾，使用 error 文字或浅底，不和普通项混在一起。
6. 二级菜单必须支持键盘方向键、Esc 关闭、点击外部关闭，并把焦点还给触发按钮。
7. 同一页面不要混用多种下拉样式；同一类入口只选一种经典样式长期复用。

### 标签规则

1. 标签用于状态和分类，不用于装饰。
2. 同类标签大小一致。
3. 重要状态用文字加颜色，不只依赖颜色。
4. 复习压力用 warning，不用 error。
5. 小型熟练度状态点可以使用语义色：记住了用成功绿，新收藏用品牌紫，有点犹豫/待复习用黄色 warning，没记住用 error；这些颜色只能作为状态提示，不能扩展成页面主题。

## 9. 页面模板

页面模板决定整体是否和谐。不要只把标准按钮和卡片随意堆在一起。

### 通用页面结构

每个页面最多包含三层：

1. 主任务区：用户当前最重要的操作。
2. 支撑信息区：解释、详情、推荐、筛选。
3. 状态区：进度、历史、提示、错误。

主任务区必须最清楚，支撑信息不能抢视觉。

### 首页导入模板

用途：首次访问、空阅读状态。

结构：

1. 品牌标题。
2. 大输入面板。
3. 一个主按钮：开始阅读。
4. 次操作：上传文件、导入链接、使用示例。
5. 首屏之后的两列等高功能展示：假名标注、句子翻译、保存生词、闪卡复习。

规则：

- 正式 Logo 可以作为品牌标题的一部分出现，尺寸要克制，不能比输入面板更抢视觉。
- 输入面板是首屏主角，不做营销型插画 hero。
- 首屏背景使用中性纸面或微灰，不用泛紫背景。
- 首屏只放品牌标题、输入面板、主按钮和文字次操作；功能卡必须在用户下滑后出现。
- 输入工作台是最明确的边界；功能展示可以使用轻边框，但边框、阴影和颜色必须弱于输入工作台。
- 首页不堆说明文案，优先让输入框和按钮自己解释用途。
- 功能预览前可以放一句轻量分区提示，例如“导入后，可以这样阅读和复习”。
- 功能预览用两列等高竖向卡片排列，画面演示占主体，功能名称固定在底部，不再放描述句。
- 预览里的紫色使用浅紫，主按钮才使用深紫。
- 四个功能卡使用同一种 surface、边框、圆角和字号。
- 不使用图标装饰次操作按钮，除非图标承担明确识别功能。

### 阅读工作台模板

用途：文章阅读、点词、查义、收藏。

结构：

1. 阅读面板。
2. 紧凑工具栏。
3. 右侧或下方详情面板。
4. 下一步行动或学习状态。

规则：

- 阅读正文最大，工具栏最轻。
- 阅读页背景、正文卡片、详情面板使用中性纸面色；紫色只用于工具选中态、焦点和少量标签。
- 正文文字使用深灰或近黑，不能用品牌色。
- 详情面板不能覆盖正文。
- 正文标注用浅底，hover/active 再增强。
- 假名、正文、词义不能重叠。

### 生词本模板

用途：保存词、筛选、复习队列。

结构：

1. 页面标题和统计。
2. 搜索和筛选。
3. 词条列表。
4. 状态标签和行操作。

规则：

- 生词本是列表工作台，不是卡片集合；页面级外框和列表大外框都应弱化。
- 顶部闪卡入口可以是独立模块，但不能被明显大卡片包住。
- 搜索、筛选、表头和词条列表属于同一工作流，用分割线和行距组织，不再额外套卡片。
- 列表优先可扫读，不做大卡片堆叠。
- 行内信息顺序固定：词、读音/释义、掌握度、来源、操作。
- 操作可以安静，但移动端必须可点。

### 语法本模板

用途：查语法、收藏句型、手动记录课堂或阅读中遇到的表达。

结构：

1. 页面标题和收藏统计。
2. 查询语法作为主入口。
3. 搜索结果或热门查阅。
4. 我的收藏。
5. 手动添加语法作为次要工作区。

规则：

- 搜索入口必须靠前，不能被大表单挤到页面下方。
- 添加语法是次要动作，表单应比搜索区更安静。
- 收藏和添加可以并列，但不能让页面变成表单堆叠。
- 热门查阅使用紧凑标签，不做大按钮组。
- 结果卡片只在展开时显示详细例句和易错点。

### 复习卡片模板

用途：闪卡、回忆、评分。

结构：

1. 进度条或计数。
2. 单张聚焦卡片。
3. 答案或提示。
4. 评分按钮。

规则：

- 一屏只做一件复习任务。
- 卡片内容居中且稳定，不因答案长短跳动。
- 评分按钮语义明确。

### 练习模块模板

用途：打字、填空、复述、语法练习。

结构：

1. 页面头：只显示模块名和当前任务类型。
2. 分段入口：选择题、复述、打字等同级练习。
3. 单个练习面板：题目或原文、输入或选择区。
4. 行动按钮。
5. 反馈区。

规则：

- 同一时间只突出一个练习面板。
- 分段入口使用浅紫选中态，不使用重色块导航。
- 反馈永远在行动之后出现。
- 不同练习共用同一套 prompt/input/action/feedback 节奏。
- 错误反馈不应让页面大幅跳动。

### 资料与历史模板

用途：推荐资料、学习历史、数据面板。

结构：

1. 页面头。
2. 筛选、分组或摘要区。
3. 列表、图表或卡片。
4. 空状态和下一步行动。

规则：

- 数据图表保持克制，颜色少而语义明确。
- 学习历史的热力图、柱状图使用紫色深浅表达强弱，不使用旧绿色主题。
- 历史记录优先显示趋势和行动建议。
- 推荐资料列表要像工具，不像营销卡片。
- 资料卡片使用浅 surface 和 1px 边框，不能做营销式大卡或重阴影。

### 设置模板

用途：配置、导出、帮助。

结构：

1. 设置分组。
2. 表单控件。
3. 保存、导出、恢复等动作。
4. 高级功能状态。

规则：

- 设置页属于二级工作区，使用“透明页面容器 + 分组卡片 + 行式设置项”的结构，避免外层大卡片再套小卡片。
- 设置页不需要营销式页面标题，也不展示“本地使用 · 无需登录”这类状态口号；需要说明时放在具体设置项里。
- 设置分组标题只保留一个紫色小标题，例如“偏好设置”“数据管理”“帮助说明”“高级功能”；不要再在下方重复一个更大的同义标题。
- 设置分组卡片使用轻边框、浅 surface、低阴影或无阴影；普通设置项使用行式分割线，不做成一张张独立小卡片。
- 表单标签、说明文字和操作按钮横向对齐；移动端再纵向堆叠。
- 普通操作按钮必须共用同一尺寸、圆角、字体和图标规则；同页按钮不要出现不同大小、不同圆角或不同图标粗细。
- 危险或覆盖性操作必须和普通操作分开，使用轻 error / caution 语义，不使用普通主按钮样式。
- 高级设置默认折叠，高级功能额度属于侧栏或次级信息，不抢主设置分组的视觉中心。
- 设置与帮助入口使用齿轮等语义图标，不使用汉字替代图标；导航图标容器、SVG 尺寸、颜色和圆角必须统一。

## 10. 布局与响应式

### 桌面

- 页面内容最大宽度优先 `1180px`。
- 阅读页使用主阅读区加详情区。
- 非阅读页使用居中的内容容器。
- 侧边导航固定，但不能压缩主内容。

### 平板

- 详情区可以下移。
- 多列列表变为两列或单列。
- 工具栏允许换行，但不能遮挡正文。

### 手机

- 顶部保留品牌栏。
- 主导航使用底部导航。
- 所有可点元素至少 `44px`。
- 表单和按钮纵向排列。
- 阅读正文先减少内边距，再考虑减少字号。
- 不允许横向溢出，长内容可以在局部区域滚动。

### 验收尺寸

每次明显 UI 修改都至少检查：

- `1440x900` 桌面。
- `1280x720` 普通桌面。
- `390x844` 手机。
- `320px` 极窄手机。

## 11. 可访问性规则

1. 键盘焦点必须清楚可见。
2. 文字对比度必须足够。
3. 图标按钮必须有可访问名称。
4. 不能只用颜色表达状态。
5. 弹窗、抽屉和菜单关闭按钮必须容易找到。
6. 触控目标至少 `44px`。
7. 动画不能影响阅读，尊重 `prefers-reduced-motion`。
8. 文本放大或窄屏时不能重叠。

## 12. 实现规则

### CSS 规则

1. 新样式优先写在 `frontend/design-system.css`。
2. 新值必须先变成 token，再进入组件。
3. 页面特有样式只写布局，不重写通用组件视觉。
4. 避免硬编码颜色、字号、圆角、阴影。
5. 避免新增 `!important`。如果是覆盖历史 CSS，要写得尽量集中。
6. 重复样式超过两处，就应该沉淀成组件或 modifier。

### 文件分工

| 文件 | 角色 |
| --- | --- |
| `DESIGN.md` | 长版设计原则与页面模板 |
| `YOMERU_UI_KIT.md` | 当前可执行的精简设计规范 |
| `frontend/typography.css` | 字体、字号、行高和字重 token |
| `frontend/design-system.css` | 颜色、通用组件和正式视觉实现 |
| `frontend/grammar-layout.css` | 语法页布局规则 |
| `frontend/styles.css` | 基础布局与历史兼容 |
| `frontend/yomeru-ui-kit.html` | token、组件和组合模式可视化验收页 |

### 新增 UI 流程

1. 先判断属于哪种页面模板。
2. 先复用现有组件。
3. 如果组件不够，先扩展组件契约。
4. 在 `frontend/yomeru-ui-kit.html` 中验证 token、单组件和组合效果。
5. 再放进真实页面。
6. 检查桌面、手机、焦点、溢出和文本重排。

## 13. 设计验收清单

每次提交 UI 改动前，逐项确认：

- 页面是否仍然是紫色 Material 3 基调。
- 是否保留了 Claude 式的编辑感：留白、克制、内容优先。
- 是否只有一个主操作最突出。
- 间距是否符合 `4 / 8 / 12 / 16 / 24 / 32 / 48` 节奏。
- 字号是否来自固定阶梯。
- 卡片是否没有明显嵌套。
- 阅读正文是否仍然是视觉主角。
- 学习状态颜色是否语义一致。
- 移动端是否无横向溢出。
- 按钮和图标目标是否至少 `44px`。
- 键盘焦点是否清楚。
- 组件展示页和真实页面是否一致。
- 同一屏的组件是否像同一个“温度”和同一种材料。

如果一个局部看起来不错，但整体不和谐，优先检查这四件事：

1. 是否偷偷新增了颜色。
2. 是否用了不在节奏里的间距。
3. 是否组件层级过多。
4. 是否主任务和辅助信息在抢视觉。

## 14. 简洁文案与交付前自检

### 文案减法标准

1. 能由标题、控件、布局或状态表达的信息，不再增加说明句。
2. 区块默认只保留标题；只有不说明会导致误操作、安全风险或关键理解错误时，才允许增加一句说明。
3. 同一含义在同一屏只出现一次。来源、状态、日期和分类不得在卡片多个位置重复。
4. 能用 5 个字表达的，不使用 10 个字；普通状态提示尽量不超过 12 个字。
5. 筛选器已经清楚表达选项时，不再增加第二套快捷筛选。
6. 空状态只保留必要状态和下一步操作，不写教程式文案。

### 卡片信息上限

1. 每张对象卡片只允许一个主标题。
2. 元信息最多 3 项，并统一为最多两种视觉样式。
3. 同一信息只显示一次；卡片不是详情页，不承载完整事实摘要。
4. 来源统一放在固定位置；可点击来源使用来源名称，而不是“查看来源”“官方信息”等泛化按钮。
5. 标签只表达分类或状态，不用于装饰。

### 人工视觉自检

自动测试通过后，交付前必须人工查看至少 `390px` 手机和 `1440px` 桌面，并确认：

- 3 秒内能找到页面主任务。
- 页面没有可删除的说明文字。
- 没有重复含义、重复日期或重复来源。
- 卡片没有多个元素同时争抢注意力。
- 卡片内信息不集中在单一角落，也没有无意义大空白。
- 不存在卡片套卡片、过多边框或不一致的胶囊样式。
- 标题、说明、来源和操作具有稳定层级。

> 自动测试通过只代表功能没有损坏；人工视觉自检通过，页面才可以交付。

## 15. 学习标签、来源与卡片交互契约

1. JLPT 等级统一使用共享 `.jlpt-level-badge` 组件；素材库、生词本和语法等页面不得各自定义不同色相或形状。
2. 未筛选的下拉框保持中性；选中具体条件后，控件本身使用浅紫背景、紫色边框和紫色文字表达激活状态。
3. 素材卡片整体可点击时，必须提供稳定的 hover、键盘焦点和轻量进入提示；不额外堆叠“进入阅读”主按钮。
4. 日语标题使用固定两行轨道，最多两行；中文说明使用固定一行轨道，同一网格的中文基线和来源基线必须对齐。
